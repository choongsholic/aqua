아래는 데모데이 발표자료 데이터 파일(slide-data.js)이야.
내 프로젝트 내용으로 채워서 완성된 slide-data.js 내용만 코드 블록으로 반환해줘.

[역할]
너는 데모데이 발표 전문 편집자야.
팀원이 제공한 날것의 정보를 받아 간결하고 임팩트 있는 발표 내용으로 재작성해줘.
단순히 옮기지 말고, 청중이 "아, 저거 필요하겠다"고 느끼도록 편집할 것.

[작성 원칙]
1. 효과 중심 — 기능이 아니라 결과를 말할 것
   X "API를 연동해 자동으로 비교 분석함"
   O "클릭 한 번으로 스펙 오차 즉시 확인"

2. 숫자 전면 — 수치가 있으면 반드시 강조
   X "검수 시간이 크게 줄었음"
   O "3시간 → 20분, 90% 단축"

3. 공감 먼저 — Problem은 청중이 "나도 그랬어" 할 언어로. 명사형 우선, 담담하게
   X "기존 방식의 비효율 존재"
   O "배포마다 수동 검수 반복 → 구조적 낭비"

4. 카드 1개 = 메시지 1개 — 부연은 발표자가 말로 전달
   X "다양한 기능을 지원하며 여러 환경에서 동작함"
   O "설치 없이 URL만으로 즉시 실행"

5. 한계는 솔직하게, 가능성으로 마무리
   X "아직 많은 한계가 있음"
   O "현재는 정적 비교만 가능 / 핵심 가치는 이미 검증됨"

6. 프로페셔널하게 — 사람이 쓴 듯 담백하게
   - 이모티콘, 느낌표 사용 금지
   - "놀라운", "혁신적인", "강력한" 등 과장 수식어 금지
   - 자신감 있되 과장 없이

[모드]
요청 유형에 따라 모드가 달라짐.

── 생성 모드 (기본) ──
[내 프로젝트 정보]가 채워진 전체 프롬프트가 오면:
- slide-data.js 전체를 생성해서 반환
- variant는 내용에 맞게 자동 선택 (아래 기준 참고)

── 수정 모드 ──
기존 slide-data.js가 첨부되고 "XX 슬라이드 수정해줘" 요청이 오면:
- 해당 슬라이드 데이터만 변경, 나머지는 그대로 유지
- variant 변경도 가능 ("타임라인으로 바꿔줘")
- 전체 slide-data.js를 반환

── 커스텀 모드 ──
기존 slide-data.js가 첨부되고 variant에 없는 자유로운 구조가 요청되면:
- override 필드를 사용해 해당 슬라이드를 custom HTML로 대체
- 디자인 규칙(아래)을 반드시 준수
- 전체 slide-data.js를 반환

override 사용 예시:
```
override: {
  problem: '<div class="slide-header" style="margin-bottom:68px;width:100%;">...'
    + '<p class="slide-tag">THE PROBLEM</p>'
    + '<h2 class="slide-title">타이틀</h2></div>'
    + '<div style="...">커스텀 콘텐츠</div>'
}
```

[커스텀 HTML 디자인 규칙]
override나 extra custom 작성 시 반드시 준수:
- CSS 변수 사용: var(--bg), var(--surface), var(--border), var(--text), var(--text-secondary), var(--accent)
- 카드: background: var(--surface); border: 1px solid var(--border); border-radius: 10px; padding: 28px 24px;
- 타이틀: font-size: calc(64px * var(--font-scale)); font-weight: 700;
- 본문: font-size: calc(28px * var(--font-scale)); color: var(--text-secondary); line-height: 1.55;
- 태그: font-size: calc(22px * var(--font-scale)); font-weight: 600; color: var(--accent); text-transform: uppercase;
- 애니메이션: class="animate", class="animate delay-1" ~ delay-8
- 컨텐츠 영역: width: min(100%, 1020px)
- inline style 사용 가능

[variant 자동 선택 기준]
팀원의 입력 형태에 따라 variant 선택:

Problem:
- 단계별 입력 (A → B → C) → 기본 (3카드 + 프로세스 플로우)
- 줄글로 문제 설명 + 세부 포인트 → focus (타이틀 + 설명 + 3카드)
- 시간순/인과 관계 → timeline (세로 타임라인)
- 하나의 핵심 문제 + 부가 설명 → single (핵심 문장 + 3카드)

Built:
- 기능 3개가 독립적 → 기본 (3카드 그리드)
- 순차적 워크플로우 → steps (세로 스텝)

How:
- 아키텍처 흐름 (입력 → 처리 → 출력) → 기본 (가로 플로우)
- 각 단계에 설명이 필요 → stack (가로 박스 카드)

[레이아웃 variant 옵션]

| 슬라이드 | variant | 특징 |
|---------|---------|------|
| problem | (기본)     | 3카드 + 프로세스 플로우 |
| problem | 'focus'    | 타이틀 + 설명 + 3카드 박스 |
| problem | 'timeline' | 좌측 세로 타임라인 |
| problem | 'single'   | 좌측 타이틀 + 좌측라인 3카드 |
| built   | (기본)     | 인용구 + 3카드 그리드 |
| built   | 'steps'    | 좌측 번호 세로 스텝 |
| how     | (기본)     | 가로 노드 플로우 + 이유 |
| how     | 'stack'    | 가로 박스 카드 + 이유(선택) |

[공통 추가 슬라이드 타입 (extra)]

| type | 특징 | 데이터 |
|------|------|--------|
| compare | 좌우 비교 카드 | left/right: { label, items[] } |
| big-number | 대형 숫자 강조 | numbers: [{ value, label }], desc |
| vertical-list | 번호 리스트 | items: [{ heading, body }] |
| image | 이미지 전체화면 | src, alt |
| cards | 카드 그리드 | items: [{ heading, body }] |
| custom | 자유 HTML | html |

[다양성 규칙 — 생성 모드에서 반드시 따를 것]
1. 전체 슬라이드 중 최소 2개는 기본이 아닌 variant 사용
2. 연속 3개 슬라이드가 같은 구조가 되지 않을 것
3. 내용과 variant가 잘 맞아야 함 (위 자동 선택 기준 참고)

[규칙]
- 텍스트는 명사형 우선. 동사가 필요할 때만 음슴체 ("~함" "~됨")
- 이미지 없으면 extra: []
- 해당 없는 선택 슬라이드는 null
- meta.accentColor / accentSecondary 는 아래 팔레트에서 선택
  [메인 accent — 푸른 계열 우선 권장]
  블루       "#3b82f6"   자동화·개발·데이터
  하늘색     "#38bdf8"   경쾌·신뢰·서비스
  형광 연두  "#a3e635"   신선함·혁신·그로스
  바이올렛   "#8b5cf6"   UX·디자인·창의
  앰버       "#F4C91D"   커머스·마케팅·주의

  [보조 accent — accentSecondary 전용]
  핑크       "#ec4899"
  로즈       "#f43f5e"
  오렌지     "#f97316"

  조합 예시: accentColor "#3b82f6" + accentSecondary "#ec4899"
             accentColor "#38bdf8" + accentSecondary "#f97316"
             accentColor "#a3e635" + accentSecondary "#3b82f6"

# [내 프로젝트 정보]

## 1. 표지

* 세션 번호와 카테고리: Session 1. Automation
* 프로젝트명: M몰 VOC Agent
* 한 줄 설명: 외부·스토어 VOC를 자동 수집·분석해 UX 개선과 마케팅 인사이트를 동시에 도출하는 AI 기반 대시보드
* 팀원 (발표자에 * 표시): 안소현*, 곽희, 송윤희

---

## 2. The Problem

* 어떤 문제를 해결했나:
  분산된 VOC를 수작업으로 수집·분석해야 하는 비효율 → UX 개선과 마케팅 의사결정이 느리고 단편적

* 기존 프로세스 또는 업무 방식:
  크롤링 요청 → 수동 수집 → 엑셀 정리 → 카테고리 분류 → 인사이트 도출 → 보고서 작성

---

## 3. What We Built

* 핵심 가치 타이틀: VOC → 실행 가능한 인사이트 자동 전환

* 핵심 기능:

  1. 통합 VOC 수집
     → 앱스토어, 플레이스토어, 블로그, 카페 등 외부 채널 VOC를 자동 수집 → 데이터 사각지대 제거

  2. 대시보드 요약
     → 평점, 부정 비율, 신규 VOC 등 핵심 지표 자동 집계 → 현황 파악 시간 단축

  3. 데이터 분포 및 이슈 분석
     → 감성/카테고리 기반 분류 + 우선순위 도출 → 무엇을 먼저 개선해야 하는지 명확화

  4. VOC 리스트 필터링
     → 소스/감성/기간/키워드 기준 탐색 → 실제 사용자 맥락 기반 문제 확인 가능

  5. 주기적 리포팅 에이전트
     → 자동 리포트 생성 → 반복 보고 업무 제거 + 지속 모니터링 체계 구축

---

## 4. How We Built
